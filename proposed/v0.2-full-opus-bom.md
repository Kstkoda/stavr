# BOM: v0.2 end-to-end on Opus, 8-hour autonomous run

**Single approval. Single agent on Opus. ~8 hours wall clock. Quota-bounded under Max plan.**

This BOM takes cowire from "design artifacts in `proposed/`" to "working v0.2 with planner + executor + first connector + Plans dashboard page" in one autonomous run. Approve once, walk away, come back to a stacked PR ready for human review.

## Goal

Land the full v0.2 substrate end-to-end: foundation files in `src/`, BOM executor module that runs approved plans, `propose_plan` MCP tool wired behind a feature flag, local-source brick installer, one concrete connector (webhook) to validate the Connector interface, the `/dashboard/plans` page rendering the food-label BOM approval card with live SSE updates, and an integration smoke test that exercises the full flow end-to-end.

## Why one BOM, not eight

Each piece depends on the previous. Stacking eight separate approvals across days means every PR gets reviewed in isolation against an incomplete substrate. One run from foundation to working flow lets Kenneth review the whole chain as a coherent unit — and lets the runner replan inside the BOM if a step surfaces something the design didn't anticipate.

- [ ] Executor module exists in `src/steward/executor.ts`, subscribes to `bom_approved`, dispatches workers per step honoring `depends_on`, persists `bom_step_started/completed/failed`, calls `planner.replan()` on failure, advances `bom.steps_done`, marks `bom.status='done'` on completion or `'failed'` on unrecoverable error.

Integration:

## Risk envelope

Allowed for the BOM's trust scope (in effect for 10 hours from approval):

- ✅ `read-only` — reading source, docs, existing DB state
- ✅ `write-local` — creating/editing files anywhere under `C:\dev\cowire\src`, `C:\dev\cowire\proposed`, the worktree's `~/.cowire/cowire.db`
- ✅ `execute` — `npm install`, `npm run build`, `npm test`, running the daemon for smoke tests, `curl` and Node scripts against `localhost:7777`
- ✅ `write-remote` — `git push` of the feature branch, opening ONE PR against `main`, adding commits to that PR

Explicitly NOT allowed (any of these triggers an interrupt-and-ask):

- ❌ `destructive` — no force-push, no rebase of shared branches, no file deletes outside `~/.cowire/tmp/` or the brick scratch dir, no schema drops on a non-test DB
- ❌ Merging the PR
- ❌ `credential` — no token rotation, no .env edits beyond adding the feature flag, no committing secrets
- ❌ `financial` — no paid-API calls outside the Max-plan quota; webhook smoke targets only public free endpoints (httpbin, jsonplaceholder)
- ❌ `external-comm` — no email, no Slack, no comments on third-party GitHub repos. Only the one PR comment on cowire itself.
- ❌ Modifying `main` directly
- ❌ Installing global npm packages outside the project
- ❌ Touching anything outside `C:\dev\cowire`

### Step 2 — Implement BOM executor (2h)

Risk: `read-only`, `write-local`, `execute`

Files:
- New: `src/steward/executor.ts`
- New: `src/steward/executor.test.ts` (unit tests)
- Edit: `src/daemon.ts` (instantiate executor at boot if planner flag is true)
- Edit: `src/event-types.ts` (ensure all BOM-step event kinds are exposed)

Build the executor:

```ts
// Shape of what to write
class BomExecutor {
  constructor(private deps: { broker, persistence, workerOrch, planner, scopeManager });

  start(): void  // subscribes to 'bom_approved' events
  stop(): void

  private async onBomApproved(payload: PayloadBomApproved): Promise<void> {
    // 1. Load BOM + active version
    // 2. Create trust_scope from bom.risk_envelope (10h ttl, action_cap = 4 * steps.length)
    // 3. Update bom.scope_id, bom.status='running', bom.started_at
    // 4. Begin dispatching steps in topological order from depends_on
  }

  private async runStep(bomId: string, stepNo: number): Promise<void> {
    // Walk dependencies → spawn worker via existing worker_orch → wait for completion
    // Capture cost, tokens, duration → persist to bom_steps
    // On failure: increment retry_count, call planner.replan() if retries exhausted
    // On replan returning willEscalateRiskClass=true: pause, emit decision_request, wait
  }
}
```

Key semantics:
- Steps run in topological order based on `depends_on`. Parallel independent steps run concurrently up to a configurable cap (default 2, per Kenneth's overnight rule of sequential single-worker — keep cap at 1 in this BOM but make it config-driven).
- A step's worker is spawned with `model = step.model`, `capability = step.capability`, scope = bom.scope_id.
- The executor itself runs on the daemon thread; workers run in child processes per existing `cc` spawner.
- On `bom_step_failed`: read failure context, call `planner.replan({ bomId, triggerStepNo, errorMessage, failedCapability, failedModel })`. If `willEscalateRiskClass` is false, swap to the new version and resume from the failed step. If true, emit `decision_request` and pause.
- On `bom_completed`: update `bom.status='done'`, `bom.ended_at`, `bom.cost_actual`. Emit `bom_completed`.
- On unrecoverable failure (replan loop exceeds 3 iterations): `bom.status='failed'`, emit `bom_failed` with reason.

Tests to write (executor.test.ts):
1. Happy path: 3-step BOM, all succeed, status transitions correctly
2. Step failure → automatic replan within envelope → resumes from replanned version
3. Step failure → replan would escalate risk class → emits decision_request, pauses
4. Daemon restart mid-BOM: on restart, finds `bom.status='running'` rows and resumes from latest incomplete step

Done when:
- Executor module exists, all 4 tests pass
- Daemon boot instantiates executor if planner flag is true
- Commit message: `feat: BOM executor with replan-on-failure and resume-on-restart`

## Failure handling

Inside the BOM scope, the runner can:

- Retry a failed step up to 3 times with the same model before calling `planner.replan()`
- Auto-replan within the risk envelope as many times as needed (the planner caps replan loops at 3 per step internally)
- Promote the model tier per profile if a step's output quality is bad (use code-execution heuristics: build fails repeatedly with the same model → try next tier)
- Skip a step ONLY if it's clearly orthogonal to the acceptance criteria (e.g., a flaky test that isn't on the path being changed). Skipping must be logged with a clear reason and surfaced in the PR description.

The runner must STOP and surface a decision request if:

- A step would require a risk class outside the envelope (force push, credential edit, external email, financial)
- Replan exhausts retries and the step is on the critical path to acceptance criteria
- The BOM scope is within 30 minutes of expiring (request extension)
- Two or more sequential commits fail to build despite the runner's fixes

## Notes for the runner (Claude Code)

- **Default model: Opus (4.8 or latest available).** Every step uses Opus. This is an explicit override of Balanced — no falling back to Sonnet without a recorded reason.
- **Why Opus everywhere:** the executor logic, replan loop, and integration test scenario design all need careful reasoning. Sonnet has tripped on similar refactors. Worth the spend (or quota) for fewer iterations.
- **Approximate cost:** $20-40 in raw API spend if metered. Under Kenneth's Max plan: quota-bounded; treat as "best effort, accept quota risk per overnight rule."
- **Worker concurrency: 1.** Single-worker sequential per overnight rule. The executor's internal concurrency cap stays config-default; do not crank it up to parallelize during this BOM.
- **Commit cadence: end of each step.** Push after every commit. The draft PR grows visibly. Kenneth can check in any time.
- **Progress signal: emit `bom_step_started` and `bom_step_completed` at every step boundary.** Use whatever event substrate is available — once step 1 lands, the new event kinds are usable; before that, fall back to generic `progress` events.
- **Trust scope: 10-hour TTL from approval.** Auto-closes at expiry. If runner finishes early, explicitly close the scope and emit a closure event.
- **If interrupted (daemon crash, network drop, runner restarted):** find the latest `bom_step_completed` event in the audit log, resume from the next pending step. Do NOT redo completed steps. Idempotency is your friend.
