# BOM: Land v0.2 foundation

**Single approval. Claude Code runs end-to-end. ~15 minutes. ~$0.50 estimated.**

This is the executable specification for landing the proposed v0.2 foundation work as a single approved plan. It's designed to be read once by Kenneth, approved, and then handed to Claude Code (or any agent runner) to execute autonomously.

## Goal

Land the proposed v0.2 foundation into `C:\dev\stavr`: SSE heartbeat fix (stability), database schema additions (BOMs, connectors, no-go list, profiles), type definitions, connector interface, and the planner skeleton behind a feature flag. Open a PR for review.

## Why this scope

These are the prerequisites for everything else (BOM planning, profile-based routing, external connectors). They're additive — nothing existing breaks. They establish the substrate without enabling user-facing behavior until follow-up PRs flip feature flags.

## Acceptance criteria (this is "done")

- [ ] `npm run build` succeeds with no new TypeScript errors.
- [ ] `npm test` passes (existing test suite green; new tests not required for this BOM).
- [ ] The `/dashboard/status` endpoint returns 200 with the daemon running locally.
- [ ] SSE connections to `/mcp/sse` no longer trigger 5-minute body-timeout errors in the switch log over a 15-minute observation.
- [ ] The new tables (`boms`, `bom_steps`, `bom_versions`, `connectors`, `no_go_list`, `profile_config`, `profile_state`) exist in `~/.stavr/runestone.db` after a fresh start. `profile_state.active_mode = 'balanced'`.
- [ ] `src/types/`, `src/connectors/`, `src/steward/planner.ts`, `src/policy/nogo.ts` compile and import cleanly.
- [ ] Feature flag `stavr.experimental.planner` exists and is `false` by default. The new planner module is loadable but not invoked.
- [ ] A PR is open against `main` with the changes, linked to this BOM.

## Out of scope (do NOT do)

- Wiring `propose_plan` as a public MCP tool — flag stays off, no users hit it yet.
- Implementing the executor that runs approved BOMs — that's a follow-up PR.
- Building any new dashboard pages or UI — backend only.
- Migrating existing trust-scope or decision flow — preserved as-is.
- Writing concrete connector implementations (Wiser, Unifi, etc.) — the interface is enough.
- Removing the existing reactive steward loop — it stays.
- Renaming anything (stavr stays as-is for now).

## Risk envelope

Allowed risk classes for this BOM's trust scope:

- ✅ `read-only` (reading existing source)
- ✅ `write-local` (creating new files in `src/`, editing `src/transports.ts`, `src/shim.ts`, `src/persistence.ts`, `src/event-types.ts`, `src/config.ts`)
- ✅ `execute` (`npm install` if needed, `npm run build`, `npm test`)
- ✅ `write-remote` (single `git commit` + `git push` of a feature branch + one PR open)

Explicitly NOT allowed:

- ❌ `destructive` (no force-push, no file deletes outside the work, no schema drops)
- ❌ `financial`
- ❌ `credential` (no token rotations)
- ❌ `external-comm` (no email, no Slack post, no comments on third-party repos)

If a step needs an action outside this envelope, the runner must stop and surface a fresh decision request. Do not proceed unilaterally.

## Steps

Each step lists: title, risk class, files affected, what "done" looks like.

### 1. Read existing source for context — `read-only`

Read these files end-to-end before touching anything:

- `src/transports.ts` (SSE route handler is here)
- `src/shim.ts` (client-side SSE consumer)
- `src/persistence.ts` (DB init + schema location)
- `src/event-types.ts` (the Zod enum to extend)
- `src/steward/loop.ts` (existing reactive loop — leave it alone, just understand the integration surface)
- `src/config.ts` or equivalent (feature flag plumbing)
- All files under `proposed/` (the spec being executed)

Don't write anything in this step. Build a mental model of where each new piece lands.

### 2. Apply SSE heartbeat fix — `write-local`

Per `proposed/sse-heartbeat-fix.md`:

- In `src/transports.ts` SSE route handler: add a `setInterval` that writes `: heartbeat <timestamp>\n\n` every 25 seconds; clear it on `req.on('close')`. Also add SSE-friendly response headers (`Cache-Control: no-cache`, `Connection: keep-alive`, `X-Accel-Buffering: no`) if not already set.
- In `src/shim.ts`: import `Agent` from `undici`, create one with `bodyTimeout: 0`, `headersTimeout: 30000`, `keepAliveTimeout: 60000`, `keepAliveMaxTimeout: 600000`. Pass it as `dispatcher` to the SSE `fetch` call.

Done when: both files compile, daemon starts, browser DevTools or `curl` on `/mcp/sse` shows `: heartbeat` lines arriving every ~25s.

### 3. Add schema migration — `write-local`

Take the contents of `proposed/001_bom_schema.sql` and integrate into `src/persistence.ts`'s `initDb()` (or equivalent init function). Place the new `CREATE TABLE IF NOT EXISTS` statements after the existing table definitions but before the seed `INSERT OR IGNORE` blocks. Add the no-go default rules and the `profile_state` seed row.

Use the existing migration pattern if there is one (look for similar `CREATE TABLE` blocks). If using raw SQL strings, keep them idempotent (the file already uses `IF NOT EXISTS`).

Done when: a fresh `~/.stavr/runestone.db` (delete and restart) has all the new tables; `SELECT * FROM profile_state` returns one row with `active_mode='balanced'`; `SELECT COUNT(*) FROM no_go_list` returns 12.

### 4. Integrate types — `write-local`

Split the contents of `proposed/types.ts` across stavr's existing structure:

- **BOM event kinds** (`bom_proposed`, `bom_approved`, `bom_rejected`, `bom_cancelled`, `bom_completed`, `bom_failed`, `bom_replanned`, `bom_edited`, `bom_step_started`, `bom_step_progress`, `bom_step_completed`, `bom_step_failed`, `bom_step_skipped`, `bom_step_promoted`, `profile_mode_switched`) and their Zod payload schemas — merge into `src/event-types.ts`'s existing Zod enum and validator.
- **Risk classes, capability tags, profile types, BOM types, payload interfaces** — new file `src/types/stavr-bom.ts`. Export everything; this is the canonical type module for v0.2 additions.
- **No-go list types + the `matchNoGo` matcher** — new file `src/policy/nogo.ts`. Import `RiskClass` from `src/types/stavr-bom.ts`.

Done when: `npx tsc --noEmit` returns clean for the whole project.

### 5. Land connector interface — `write-local`

Take `proposed/connector.ts` and place it at `src/connectors/connector.ts`. Adjust the import of `RiskClass` and `CapabilityTag` to come from `src/types/stavr-bom.ts`.

Create an empty `src/connectors/index.ts` that re-exports the interface plus a stub `ConnectorRegistry` implementation (in-memory `Map<string, Connector>`, no persistence yet — that's a follow-up).

Done when: importing `Connector` and `ConnectorRegistry` from `src/connectors` compiles. The stub registry's `register`, `get`, `list`, `listByKind`, `allCapabilities` methods all work for an empty set.

### 6. Land planner skeleton (feature-flagged) — `write-local`

Take `proposed/steward-planner.ts` and place it at `src/steward/planner.ts`. Adjust imports to use real stavr types (the file currently has stub `EventEmitter`, `Persistence`, `LlmProvider` type aliases at the top — replace those with the real stavr counterparts from existing modules).

Add a feature flag `stavr.experimental.planner` (default `false`) to `src/config.ts`. The flag is checked at planner instantiation time; if false, don't instantiate, log "planner disabled".

Done when: planner compiles, runs end-to-end in a unit-test-style invocation against an in-memory database when flag is forced true, and is not invoked in any code path when flag is false (no `new StewardPlanner(...)` outside the flag guard).

### 7. Build and smoke test — `execute`

```sh
npm run build
npm test
```

If TypeScript errors surface, fix them in place (likely missing imports, type mismatches between proposed types and real stavr types). Don't relax types — fix the actual mismatch.

Then manual smoke:

```sh
node dist/cli.js daemon start
# in another shell:
curl -N http://127.0.0.1:7777/mcp/sse | head -20
# should show MCP init + `: heartbeat` lines
```

Stop the daemon. Delete `~/.stavr/runestone.db`. Start it again. Verify all new tables exist via `sqlite3 ~/.stavr/runestone.db ".tables"`.

Done when: build green, tests green, fresh DB has new tables, SSE shows heartbeats.

### 8. Commit + open PR — `write-remote`

Create a feature branch: `feature/v0.2-foundation`.

Recommended commit structure (preserves bisectability):

1. `fix: SSE heartbeat + undici bodyTimeout to eliminate 5min reconnect cycle`
2. `feat(db): add boms, connectors, no-go, profile tables (additive, idempotent)`
3. `feat(types): risk classes, capability tags, profile modes, BOM types, no-go matcher`
4. `feat(connectors): Connector interface + stub registry`
5. `feat(steward): planner skeleton behind experimental.planner flag`
6. `docs: rewrite README around the four pillars; add POSITIONING; v0.2 foundation BOM`

Push to remote. Open a PR against `main` with this description:

```
Lands the v0.2 foundation per proposed/v0.2-foundation-bom.md.

Substrate only — no user-facing behavior changes until follow-up PRs flip feature flags.

Acceptance criteria from the BOM:
- [x] Build + tests green
- [x] SSE heartbeats live; no body-timeout errors over 15min observation
- [x] New tables present on fresh DB; profile_state.active_mode='balanced'
- [x] All new types compile
- [x] stavr.experimental.planner flag exists, default false

Follow-ups (not in this PR):
- Executor that runs approved BOMs
- propose_plan MCP tool
- Concrete connector implementations
- Dashboard pages for Plans / Decide / Topo / Streams / Kit
- External brick sources (see proposed/external-brick-sources.md)
```

Do NOT merge the PR. Stop at "PR open" and surface the PR URL.

Done when: PR is open, CI passes, URL is captured in the BOM completion event.

## Failure handling

If any step fails:

1. **TypeScript errors after step 4 or 5**: re-read the type files, fix imports, re-run `npx tsc --noEmit`. If genuinely stuck after one retry, surface as a decision request asking Kenneth which way to resolve the mismatch — don't guess.
2. **Tests fail in step 7**: read the failure output, identify whether it's a pre-existing flake or caused by our changes. If caused by our changes, fix. If pre-existing, document in the PR description but proceed.
3. **Smoke test fails (heartbeats not appearing)**: re-read step 2, check the interval is unref'd if needed, check the response writes aren't being buffered by Express's compression middleware. If still stuck after one retry, surface as decision request.
4. **Step 8 push fails**: likely auth or branch protection. Surface as decision request — never `--force` or rewrite history without explicit approval.
