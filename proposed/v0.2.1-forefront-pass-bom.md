# BOM: v0.2.1 Forefront Pass

**Single approval. Six checkpoints. ~10-14h Opus run. ~$8-12 estimated.**

Lands every remaining v0.2 audit finding AND advances stavr to 2026 state-of-practice on the four axes external research flagged: transport, sandboxing, traceability, and the lego principle for brick lifecycle. By the end of this BOM, stavr is on the forefront — not chasing the moving floor.

Six checkpoints, each merge-safe. The runner pauses for human approval between checkpoints; within a checkpoint, work is autonomous.

## Why this scope

The v0.2 audit found 1 blocker, 3 majors, 5 minors, 4 infos. Architectural validation found:
- Standalone `SSEClientTransport` is `@deprecated` in the MCP SDK; the ~June-2026 spec revision consolidates around Streamable HTTP.
- The brick layer is the 2026 anti-pattern *unless* every in-house brick declares its replacement contract and supports swap. **Lego principle:** every brick is a placeholder.
- Node `--permission` flag went stable in v23.5.0 (late 2025). Loading factory functions from disk with zero isolation is below the 2026 floor.
- OTel GenAI semconv is the consensus traceability adapter even while experimental.

Doing each as a separate small PR is correct; bundling under one approved plan is fast.

## Goal

By end of this BOM:
- Every audit finding (blocker, majors, minors, infos) closed or explicitly accepted with rationale.
- Every in-house brick declares its `bridges_to` target. Dashboard surfaces "this brick can be replaced" when a matching MCP server is discovered.
- Shim speaks Streamable HTTP. SSE remains one release with `Sunset:` header + deprecation log.
- Each `brick.exec()` runs in a `worker_threads.Worker` with permissions aligned to its trust-scope risk class.
- Daemon emits OTel GenAI spans alongside the SQLite event stream when an OTLP endpoint is configured.

## Out of scope

- Replacing the SQLite event store (keep — see architecture validation memo 2026-05-15).
- LangGraph / LiteLLM adoption (re-evaluate post-v0.3).
- Multi-machine sync (privacy-preserving local-first stays the v0.x posture).
- v0.3 shared-memory BOM (separate file, separate approval — features, not infrastructure).
- Custom OAuth provider implementations beyond the standard flows (`oauth2_client_credentials`, `oauth2_authorization_code`).

## Risk envelope

- ✅ `read-only`, `write-local`, `execute` (npm build/test), `write-remote` (one push + one PR per checkpoint).
- ❌ `destructive` (no force-push, no schema drops outside the migration pattern), `financial`, `credential` rotation, `external-comm` (no Slack/email).
- New for this BOM: ✅ `worker-spawn` (creating `worker_threads.Worker` instances at runtime — internal to the daemon, no shell escape).

If a step needs anything outside this envelope: stop and surface a fresh decision request.

## Checkpoint 3 — Lego principle: replaceable bricks (~2h)

Branch: `feature/v0.3-lego-replaceable-bricks` off main after C2.

The principle in one sentence: **every in-house brick is explicitly a placeholder; the manifest declares what would replace it, and when that replacement becomes available the dashboard surfaces a one-click swap.**

- **3.1** Extend `stavr-brick.json` schema with optional field:
  ```
  bridges_to: {
    type: "mcp" | "github" | "npm" | "none",
    target: string,           // e.g. "mcp:official-unifi", "github:owner/wiser-mcp"
    rationale: string,        // why this brick exists today; what triggers retirement
  }
  ```
  Update `parseBrickManifest` validator. `none` is valid for primitives (webhook, scheduler) that aren't bridging a specific service.
- **3.2** Stamp existing in-house bricks with their `bridges_to` values. Webhook → `{type:"none", rationale:"generic HTTP primitive"}`. Sample → `{type:"none", rationale:"reference implementation"}`. Future Wiser/Unifi/Roblox/Unity bricks declare upstream MCP targets in their manifests when written.
- **3.3** Add `brick_replaceable` event kind. Emit when the discovery probe (next step) finds an MCP server matching a brick's `bridges_to.target`.
- **3.4** Add MCP discovery probe: scheduled hourly check against configured MCP registries (initial set: Klavis, mcp.run, Smithery — config-driven so new registries plug in). Cache results. For each registered brick with `bridges_to.type==="mcp"`, check whether a matching server exists.
- **3.5** Dashboard: add "Replaceable" status indicator to brick cards on the Toolkit page. Single-click swap workflow: open confirm dialog → install replacement (via existing installer or MCP server registration) → deactivate in-house brick → emit `brick_swapped` with `from`, `to`, `at` payload.
- **3.6** ADR-025: lego principle for brick lifecycle.

Done when: every brick's manifest carries `bridges_to`, the discovery probe runs hourly, the dashboard surfaces replaceable bricks, and ADR explains the philosophy.

Done when: shim runs on Streamable HTTP by default, SSE still works as a fallback for one release, ADR documents the timeline.

## Checkpoint 5 — Brick sandboxing (~3h)

Branch: `feature/v0.3-brick-sandbox` off main after C4.

- **5.1** Daemon wrapper: `node --permission --allow-fs-read=$STAVR_HOME,$BRICKS_ROOT --allow-fs-write=$STAVR_HOME --allow-net=127.0.0.1:7777,<brick target allowlist>`. Update the daemon entry script and systemd/launchd templates. Document in `docs/operating.md` how to add net allowlist entries when a new brick needs an outbound host.
- **5.2** Refactor `connector.exec()` invocation path: spawn each exec in a `worker_threads.Worker`. Structured-clone message bus for input/output. Worker pool with `resourceLimits: { maxYoungGenerationSizeMb: 64, maxOldGenerationSizeMb: 256 }` per worker — adjustable per brick via manifest.
- **5.3** Map trust-scope risk classes → worker permission allowlist:
  - `read-only` → no net, no fs-write
  - `write-local` → fs-write within data-dir only
  - `external-comm` → net within brick's declared target hosts
  - `execute` → worker can spawn child processes within the data-dir
  - `destructive` / `financial` / `credential` → require explicit per-action prompt; not granted to worker preemptively
- **5.4** Brick crash isolation test: register a brick whose exec throws and uses `process.exit(1)` to be hostile. Assert daemon survives, executor records `bom_step_failed`, retry counter increments, no other workers affected.
- **5.5** ADR-026: brick sandboxing model — permission flags + worker isolation + capability-to-permission mapping.

Done when: every `brick.exec()` runs in an isolated worker, daemon survives malicious brick scenarios, permission model documented.

## Checkpoint 6 — OTel GenAI adapter (~1.5h)

Branch: `feature/v0.3-otel-adapter` off main after C5.

- **6.1** Add deps: `@opentelemetry/api`, `@opentelemetry/sdk-trace-node`, `@opentelemetry/exporter-trace-otlp-http`, `@opentelemetry/semantic-conventions`.
- **6.2** New module `src/observability/otel-adapter.ts`: subscribes to broker, maps domain events → GenAI spans per the experimental semconv. Key mappings:
  - `bom_proposed` → `gen_ai.task.start` (root span per BOM)
  - `bom_step_started` → `gen_ai.action.start` (child of BOM span)
  - `bom_step_completed` / `bom_step_failed` → end action span with status
  - `bom_completed` / `bom_failed` → end task span
  - `trust_scope_granted` → span event attached to BOM span
  - `decision_request` / `decision_responded` → human-in-loop span events
- **6.3** Config: `stavr.observability.otlp_endpoint` (default `null` = disabled). When set, the adapter starts on daemon boot. Endpoint format: standard OTLP/HTTP, points at Langfuse / Phoenix / Datadog / a local Jaeger.
- **6.4** Integration test: configure OTLP endpoint pointing at a mock OTLP collector (existing patterns: a tiny in-process HTTP server matching the OTLP/HTTP spec). Run one BOM end-to-end, assert spans arrive with trace_id continuity (root BOM → action steps), span attributes carry brick_id + risk_class + cost.
- **6.5** ADR-027: traceability via OTel adapter — rationale for keeping SQLite as system-of-record AND emitting OTel as adapter (lossy but vendor-friendly).

Done when: OTLP exporter is optional, off by default, and when enabled produces well-formed GenAI spans that render correctly in at least one of Langfuse/Phoenix/Datadog.
