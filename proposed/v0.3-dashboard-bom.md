# BOM: v0.3 Dashboard Rebuild

**Single approval. Ten checkpoints. ~12-18h Opus run. ~$10-14 estimated.**

Builds the full multi-page stavr dashboard at `http://127.0.0.1:7777/dashboard` — Home, Topology, Streams, Plans, Decide, Toolkit, Capabilities, Settings — in Dark 2.0 with the food-label visual grammar we designed. Replaces the current single-page shell with a proper navigation surface that surfaces what stavr is doing without you needing to read the event stream.

Closes the gap between "we designed beautiful dashboards" and "they exist in the product."

## Why this scope

The v0.2 daemon ships a single `/dashboard` HTML page + 11 JSON APIs. Everything else was mockup-only in chat sessions. The forefront BOM (v0.2.1) closes audit findings + infrastructure modernization but leaves the dashboards untouched except for one Plans page polish.

This BOM does the UI work as a coordinated rebuild. Single approval, one design language, one nav surface. Each checkpoint lands a complete page so you can preview as work progresses.

## Goal

By end of this BOM, navigating to `http://127.0.0.1:7777/dashboard` gives:

- **Home** — at-a-glance daemon health, active BOMs, recent decisions, quick actions
- **Topology** — Unifi/FR24-style ops control center with worker positions, time-scrubber for history rollback, click-to-execute (tail/inspect/terminate)
- **Streams** — live multi-pane terminal view for 10-20 concurrent workers with filters + search
- **Plans** — food-label BOM approval cards (What / Risk / Reversible / Cost) with full design parity to the mockups
- **Decide** — food-label decision cards for the same approval surface
- **Toolkit** — ESB bus visualization with LEGO/DUPLO bricks (internal below the bus, external above), click-to-edit per-brick config including auth/oauth/api-keys
- **Capabilities** — Lego baseplate showing what each profile mode (Turbo/Balanced/Eco) unlocks
- **Settings** — profile mode switcher, trust scope manager, no-go list editor, brick installer

All pages share Dark 2.0 styling, the food-label visual grammar, the brick-as-color-coded-shape convention (AI=purple, internal-AI=yellow, MCP=blue, steward=red, connectors=orange/green per kind), and live SSE-driven updates.

## Out of scope

- Mobile responsive layouts (desktop-first; mobile is a v0.4 concern)
- Multi-user authentication on the dashboard (loopback-only stays the v0.3 posture)
- A "build your own BOM" drag-and-drop planner UI (planner is steward-driven, not user-built)
- Replacing the JSON APIs — `/dashboard/plans/list` etc. stay as the data plane; pages consume them
- Internationalization (English-only for v0.3)
- Theming beyond Dark 2.0 (no light mode for v0.3)

## Risk envelope

- ✅ `read-only`, `write-local`, `execute` (build + test), `write-remote` (one push + one PR per checkpoint).
- ❌ `destructive`, `financial`, `credential` rotation, `external-comm`.

## Acceptance criteria

Per checkpoint, inline. Each checkpoint ends with: build green, tests green, PR open with screenshots of the new page in the description.

## Design references

All design decisions captured in memory file `project_cowire_dashboard_modes.md`. Key invariants:

- **Color coding**: external AI = purple, internal AI = yellow, MCP utilities = blue, steward = red, connectors = orange (above bus) / green (below bus).
- **Food label visual grammar**: 4-cell card with What / Risk / Reversible / Cost. Reused for BOMs, decisions, trust scopes.
- **ESB framing for Toolkit**: steward is the bus (red), external above (purple/orange), internal/local below (yellow/green/blue).
- **No connecting lines between bricks** — they're either registered or not. Lego/DUPLO don't have wires.
- **Click on any brick** opens the inspector floating panel with config fields per its `config_schema`.
- **Three profile modes**: Turbo (Opus all the way), Balanced (Sonnet + Opus on hard steps), Eco (Sonnet/Haiku, fail-fast).
- **Time scrubber** on Topology page lets you scroll back through the event stream and see the topology at any prior moment.

## Checkpoint 1 — Foundation: routing, shell, design tokens, components (~2h)

Branch: `feature/v0.3-dashboard-foundation` off main.

**1.1** Refactor `src/dashboard-html.ts` from one-page shell into a navigation surface with route-based content loading. Use server-side templating for the shell + nav, client-side JS to swap content via fetch + hash routing (`#/home`, `#/topology`, etc.).

**1.2** Add `src/dashboard/` directory with per-page module structure:
```
src/dashboard/
  shell.ts            // top nav + sidebar
  tokens.css.ts       // design tokens (Dark 2.0 CSS custom properties)
  components/
    food-label.ts     // 4-cell What/Risk/Reversible/Cost card
    brick.ts          // SVG brick with color-coded type
    scrubber.ts       // time-scrubber slider
    pill.ts           // status pills
    inspector.ts      // floating right-side panel
  pages/
    home.ts
    topology.ts
    streams.ts
    plans.ts
    decide.ts
    toolkit.ts
    capabilities.ts
    settings.ts
```

**1.3** Add `mountDashboardRoutes` per-page registration: `/dashboard/home`, `/dashboard/topology`, etc. Each returns its page HTML rendered with the shell. `/dashboard` → 302 to `/dashboard/home`.

**1.4** Define Dark 2.0 design tokens as CSS custom properties:
```
--bg-base: #0a0a0f
--bg-surface: #14141a
--bg-elevated: #1c1c24
--text-primary: #e8e8f0
--text-secondary: #8a8a96
--accent-ai-external: #a78bfa  (purple)
--accent-ai-internal: #facc15   (yellow)
--accent-mcp: #60a5fa            (blue)
--accent-steward: #ef4444        (red)
--accent-connector-above: #fb923c (orange)
--accent-connector-below: #4ade80 (green)
--risk-low: #4ade80
--risk-medium: #facc15
--risk-high: #ef4444
```

**1.5** Build the shared `food-label.ts` component. Returns HTML for the 4-cell card. Input: `{ what, riskClass, reversible, costUsd }`. Same component used by Plans, Decide, Topology hover-cards.

**1.6** Build the shared `brick.ts` SVG component. Input: `{ id, kind, displayName, position, status }`. Output: color-coded SVG rounded rect with stud dots on top.

**1.7** Build the `inspector.ts` floating panel. Right-side, fixed position, slides in when an item is clicked. Renders a form from the item's `config_schema`. Used by Toolkit, Topology.

**1.8** Build `tests/dashboard/shell.test.ts` and `tests/dashboard/components/*.test.ts` for the foundation. Snapshot tests for component output.

Done when: navigating to `/dashboard` redirects to `/dashboard/home` which renders an empty shell with the nav surface and Dark 2.0 styling. Components are importable + tested. All existing dashboard tests still pass.

## Checkpoint 3 — Plans page food-label rebuild (~2h)

Branch: `feature/v0.3-dashboard-plans` off main after C2.

**3.1** Replace the current simplified Plans card with the full food-label design. Each BOM row in the list is a 4-cell card:
- **What**: BOM title + brick id glyphs + step count
- **Risk**: highest risk class in envelope, color-coded
- **Reversible**: ✓ Yes / ⚠ Partial / ✗ No (computed from steps' risk classes)
- **Cost**: estimated_cost_usd + model mix indicator (Sonnet / Opus mix)

**3.2** Click a card → expanded view with full step list, dependencies graph (SVG node-link), trust scope it'll create, approve/reject buttons.

**3.3** Approve button → POSTs to existing `/dashboard/plans/:bomId/respond` (no API change). On success → animates card to "running" state + jumps to Topology.

**3.4** Allowed / Will-ask-first split in the Risk cell: visualize the intersection of the BOM's risk envelope and the active no-go list as "✓ approved without re-prompt" vs "⚠ will pause for each".

**3.5** Tests: update `tests/dashboard-plans.test.ts` to cover food-label rendering + Allowed/Will-ask split logic.

Done when: Plans page is the design we mocked. Audit minor #1 closed.

**4.5** Tests: `tests/dashboard/decide.test.ts` — render open decisions, click option emits `decision_responded`, countdown updates.

## Checkpoint 5 — Topology page (~3h)

Branch: `feature/v0.3-dashboard-topology` off main after C4.

The most complex page. SVG-based ops control center.

**5.1** SVG canvas. Daemon node at center (red circle). Connectors above the bus, workers below — same ESB framing as Toolkit but with live state.

**5.2** Worker nodes color-coded by status: green=idle/healthy, yellow=running, red=crashed, gray=terminated. Pulse animation on currently-active workers.

**5.3** Connector nodes color-coded by kind: external AI (purple), MCP (blue), connector-above (orange), connector-below (green), internal AI (yellow).

**5.4** Each node click → opens the floating Inspector panel with: live status, recent events, log tail, actions (Tail / Inspect / Terminate / Restart).

**5.5** Time scrubber along the bottom. Drags scrub the rendered topology backward through history. Snapshots taken every `event.created_at` boundary; topology re-renders to match the state at that moment. Released → snaps back to live.

**5.6** Live updates via `/dashboard/stream` SSE. Each event with `kind` matching `bom_step_*`, `worker_*`, `trust_scope_*` triggers a topology refresh.

**5.7** Right-side: list of in-flight BOMs grouped by trust scope, with mini food-label.

**5.8** Tests: `tests/dashboard/topology.test.ts` — assert SVG structure renders given a fixed event log fixture, scrubber position changes the rendered nodes, click → inspector opens.

Done when: this is the page you'd leave open on a second monitor while stavr runs overnight.

## Checkpoint 6 — Streams page (~2h)

Branch: `feature/v0.3-dashboard-streams` off main after C5.

**6.1** Multi-pane terminal view for live worker output. Up to 20 panes in a responsive grid (4-wide on desktop).

**7.7** Tests: `tests/dashboard/toolkit.test.ts` — render existing connectors as bricks, click → inspector renders correct fields, save calls applyConfig.

## Failure handling

Per checkpoint: if any step fails after one retry, runner stops and surfaces a decision request. No checkpoint auto-proceeds. Each PR can be merged or rejected independently.

If a checkpoint reveals scope that wasn't anticipated (e.g. C7 toolkit's brick install flow exposes a missing installer behavior), surface as decision request — don't expand scope unilaterally.

## Notes for the runner

- **Default model**: Sonnet for HTML/CSS/component code, Opus for the Topology page SVG + time-scrubber logic + design decisions.
- **Estimated wall-clock**: 12-18h across ten checkpoints. C5 (Topology) is the longest single block.
- **Estimated cost**: ~$10-14 in Balanced mode.
- **Idempotency**: every step restartable. Checkpoint commits = clean rollback boundaries.
- **Tech stack constraint**: server-rendered HTML + vanilla JS (no React/Vue/Svelte). Express templates already in the codebase. Why: stavr is local-first, no build step on the client, dist/ stays small.
- **Live update channel**: `/dashboard/stream` SSE already exists and is plumbed to the broker. Pages subscribe with `EventSource('/dashboard/stream')` for delta updates.
- **Screenshots**: each checkpoint PR description includes a screenshot of the new page (runner uses headless Chrome via Puppeteer or playwright dev dep).
- **Approval scope expiry**: 24h after approval. If not done in 18h, surface extension request.
