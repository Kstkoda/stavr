# BOM: v0.4 Steward Scheduler — BOM backlog and cross-BOM orchestration

**Single approval. Seven checkpoints. ~10-14h Opus run. ~$8-12 estimated.**

Promotes the Steward from a **reactive event handler** (today: subscribes to `bom_approved`, runs the BOM) into a **scheduler** that orchestrates a real backlog: priority-ordered, capacity-gated, dedupable, surfaced to the user one-at-a-time for approval.

Closes the architectural gap surfaced 2026-05-15: when CC autonomously proposes 7+ BOMs back-to-back during a long run, today they flood the Decide page in arrival order with no throttling, no prioritization, no capacity ceiling. v0.4 fixes that.

## Why this scope

v0.2 shipped the substrate (BOM types, executor, trust scopes). v0.3 ships the dashboard surface. v0.4 ships the **scheduling discipline** that makes long autonomous runs sustainable.

Without a scheduler:
- Autonomous CC runs flood the Decide page with N BOMs in unpredictable order.
- No way to express "do the blocker BOM before the polish BOM."
- One BOM can hog all execution capacity; nothing fair across multiple in-flight BOMs.
- The Steward can't dedupe near-identical BOMs (CC proposing the same fix twice).
- Restart-resume only picks up where a SINGLE BOM left off; the backlog itself isn't restart-aware.

With the scheduler:
- Steward becomes the orchestrator the original design intended.
- Plans page renders a real queue ("3 of 5 — awaiting approval").
- Settings page exposes capacity + default-priority knobs.
- BOMs CC proposes during a run get throttled into a manageable stream.
- Failures + retries get bounded cooldowns, not infinite loops.

## Goal

By end of this BOM:

- A new `src/steward/scheduler.ts` module runs a periodic loop (default 5s) that pulls from the backlog, applies priority + capacity + dedupe rules, and surfaces one BOM at a time for approval.
- The `boms` table gains `priority`, `enqueued_at`, `surfaced_at`, `dedupe_key` columns.
- Five new event kinds: `bom_enqueued`, `bom_surfaced`, `bom_deferred`, `bom_capacity_reached`, `bom_duplicate_skipped`.
- The Plans page (from v0.3 C3) renders queue position + priority + deferred reason.
- The Settings page (from v0.3 C9) exposes capacity limits + default priority + dedupe window.
- A capacity-limit setting throttles execution: at most N BOMs running at once (default 1, matching today's behavior; upgradeable to 2-3 once trust-scope-aware scheduling stabilizes).
- A priority-aware approval surface: the Decide page shows the highest-priority surfaced BOM at top.
- Restart-aware backlog: daemon restart preserves backlog state and resumes scheduling without re-surfacing already-decided BOMs.

## Out of scope

- Cross-machine scheduling (federation — that's a v1.0 concern).
- Cost-aware scheduling (running cheap BOMs first to preserve Anthropic quota — that's a v0.5 concern bundled with the cost-routing work).
- ML-based priority inference. Priority is operator-set or BOM-author-set; the scheduler doesn't try to guess.
- Multi-user priority (one user's "critical" outranking another's "polish" — single-user system for v0.x).

## Risk envelope

Same as v0.3:
- ✅ `read-only`, `write-local`, `execute` (build + test), `write-remote` (one push + one PR per checkpoint)
- ❌ `destructive`, `financial`, `credential` rotation, `external-comm`

If scheduling logic ever needs to autonomously cancel a running BOM: that's CONFIRM-tier via existing `await_decision` infrastructure, not a destructive op.

## Checkpoint 1 — ADR + schema migration (~1.5h)

Branch: `feature/v0.4-scheduler-schema` off main.

**1.1** Write `adr/029-steward-as-scheduler.md`:
- The decision: Steward goes from reactive event handler to scheduler with explicit backlog
- The why: autonomous runs flood the approval surface; no priority; no capacity
- The how: periodic loop, priority queue, capacity gate, dedupe window
- The alternative considered: keep reactive, add priority field — rejected because reactive can't enforce capacity or dedupe without becoming a partial scheduler anyway
- The trade-off: adds 200-300 lines of stateful code; gains a coherent orchestration model

**1.2** Schema migration in `app/scripts/init-db.ts` (the actual migration site — Drizzle is bypassed). Add columns to `boms`:
```sql
ALTER TABLE boms ADD COLUMN priority INTEGER NOT NULL DEFAULT 5;       -- 0 = highest, 9 = lowest
ALTER TABLE boms ADD COLUMN enqueued_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP;
ALTER TABLE boms ADD COLUMN surfaced_at TIMESTAMP NULL;
ALTER TABLE boms ADD COLUMN dedupe_key TEXT NULL;
ALTER TABLE boms ADD COLUMN deferred_reason TEXT NULL;   -- 'capacity' | 'dedupe' | 'priority' | NULL
ALTER TABLE boms ADD COLUMN deferred_until TIMESTAMP NULL;
CREATE INDEX IF NOT EXISTS idx_boms_priority_enqueued ON boms(priority ASC, enqueued_at ASC) WHERE status IN ('proposed', 'deferred');
CREATE INDEX IF NOT EXISTS idx_boms_dedupe ON boms(dedupe_key) WHERE dedupe_key IS NOT NULL;
```

Schema migration must be idempotent (use `IF NOT EXISTS` / catch-and-skip). Use the SQLite table-rebuild pattern if needed since SQLite doesn't support all `ALTER TABLE` variants.

**1.3** Add `'deferred'` to the BOM status enum (existing values: `proposed`, `running`, `done`, `failed`, `cancelled`). Update `src/types/stavr-bom.ts`.

**1.4** Tests:
- Schema applies on fresh DB; idempotent on re-init
- Index exists and is used by `EXPLAIN QUERY PLAN` for the scheduler's typical query

Done when: a new DB has the new columns, and `SELECT * FROM boms WHERE status='proposed' ORDER BY priority, enqueued_at` uses the index.

## Checkpoint 2 — New event kinds (~1h)

Branch: `feature/v0.4-scheduler-events` off main after C1.

**2.1** Extend `src/event-types.ts` with five new event kinds:

```ts
'bom_enqueued'           // BOM joined the backlog
  payload: { bom_id, priority, enqueued_at, dedupe_key?, source: 'user' | 'cc' | 'steward' }

'bom_surfaced'           // Steward selected a BOM for approval
  payload: { bom_id, surfaced_at, queue_position, total_backlog }

'bom_deferred'           // Steward held a BOM back (capacity / dedupe / priority)
  payload: { bom_id, reason: 'capacity' | 'dedupe' | 'priority', deferred_until?, dedupe_target_bom_id? }

'bom_capacity_reached'   // Scheduler hit the max-concurrent limit
  payload: { running_count, capacity_limit, deferred_bom_ids: string[] }

'bom_duplicate_skipped'  // Scheduler detected a duplicate via dedupe_key
  payload: { bom_id, duplicate_of_bom_id, dedupe_key }
```

**2.2** Update tests in `tests/event-flow.test.ts` to cover each new event kind. Validate Zod schemas; ensure they round-trip through `broker.publish` and `get_events`.

**2.3** Update `docs/event-taxonomy.md` with the new kinds and their semantic ordering relative to existing ones (e.g. `bom_proposed` now → optional `bom_enqueued` → `bom_surfaced` → `bom_approved`).

Done when: 5 new event kinds are loaded, Zod-validated, persisted, queryable, documented.

## Checkpoint 3 — Scheduler module (~3h)

Branch: `feature/v0.4-scheduler-loop` off main after C2.

The core piece. New file: `src/steward/scheduler.ts`.

**3.1** Implement `StewardScheduler` class:
```ts
class StewardScheduler {
  constructor(opts: {
    store: SchedulerStore,            // get/update BOM rows
    events: PlannerEventEmitter,      // emit bom_*  events
    config: SchedulerConfig,          // capacity, dedupe window, interval
    intervalMs?: number,              // default 5000
  });
  start(): void;
  stop(): void;
  // Public probe for tests + dashboard:
  status(): { running: number; surfaced: number; backlog: number; capacity: number };
}
```

**3.2** The loop body (every `intervalMs`):
1. **Dedupe sweep**: for each `status='proposed'` BOM with `dedupe_key`, check if another BOM with the same `dedupe_key` is `running` or `surfaced` in the last `dedupe_window_sec`. If yes → mark `status='deferred'`, `deferred_reason='dedupe'`, emit `bom_duplicate_skipped`.
2. **Capacity check**: count `status='running'` BOMs. If >= `capacity_limit`, emit `bom_capacity_reached` (throttled — once per interval, not spammed).
3. **Priority pick**: if capacity available AND no `surfaced` BOM awaiting approval, pick the next `proposed` BOM by `(priority ASC, enqueued_at ASC)`. Mark `surfaced_at = NOW()`, emit `bom_surfaced`.
4. **Re-queue deferred**: for each `deferred` BOM with `deferred_until <= NOW()`, flip back to `proposed`. Emit `bom_enqueued` again.

**3.3** Wire into `src/steward/v02-wiring.ts`: instantiate the scheduler alongside the executor when `experimental.scheduler` flag is on (default true in v0.4). Hook stop() into daemon shutdown.

**3.4** Add `experimental.scheduler` flag to `stavr.yaml` schema with default `true`. Document in `docs/configuration.md`.

**3.5** Tests in `tests/steward/scheduler.test.ts`:
- Empty backlog: loop runs cleanly, no events emitted
- Single BOM, capacity 1: surfaced immediately on first tick
- Two BOMs, capacity 1, one running: second one not surfaced
- Two BOMs same `dedupe_key`: second deferred with reason='dedupe'
- Priority order: BOM with priority 1 surfaces before BOM with priority 5
- Deferred-until expiry: re-queues correctly on next tick
- Stop(): loop stops cleanly, no leaked intervals

Done when: scheduler runs the periodic loop, emits the right events, surfaces BOMs in priority order, respects capacity.

## Checkpoint 4 — Backlog enqueue path + capacity-aware approval (~1.5h)

Branch: `feature/v0.4-scheduler-enqueue` off main after C3.

**4.1** Update `bom_proposed` event handling: when a BOM is proposed via `propose_plan` MCP tool, also emit `bom_enqueued` with the proposed priority (default 5), source (`'cc'` or `'user'`), and optional `dedupe_key`.

**4.2** Extend `propose_plan` MCP tool to accept optional `priority` (0-9) and `dedupe_key` (string) args. Default priority=5, dedupe_key=null.

**4.3** Capacity-aware approval surface: the existing `bom_approved` flow now goes through the scheduler — instead of executor reacting directly to `bom_approved`, the scheduler updates the BOM to `status='running'` only if capacity allows. If not, defer with `reason='capacity'`. When a running BOM finishes (`bom_completed` / `bom_failed`), scheduler picks the next deferred one.

**4.4** Restart-aware: on daemon boot, scheduler's first action is to scan for `status='running'` rows (the executor's resume logic handles those), and `status='surfaced'` rows older than 1h get rolled back to `'proposed'` (they were surfaced but user never decided — re-surface with priority bump?).

**4.5** Tests:
- propose → enqueued event has correct priority + source
- bom_approved with capacity full → deferred, not running
- Running BOM completes → next deferred BOM surfaces
- Restart with one running + two surfaced → running continues, surfaced get re-queued

Done when: the full ProposePlan → Enqueue → Surface → Approve → Run pipeline respects capacity.
