# Contributing to Stavr

This document is for someone making a change to Stavr — adding an adapter, adding an event kind, fixing a bug, or refactoring. If you only want to understand the architecture, read [`ARCHITECTURE.md`](./ARCHITECTURE.md) first.

---

## Repo layout

```
.

│   ├── event-types.ts           Zod schemas for every event kind
│   ├── paths.ts                 default file locations (~/.stavr/...)
│   ├── persistence.ts           SQLite event/decision/session store
│   ├── server.ts                MCP server assembly + core tool registration

```

`dist/` (generated by `npm run build`) and `node_modules/` are git-ignored.

## Development setup

Requires Node ≥ 20. Native module `better-sqlite3` will compile against your Node version on first install — if you change Node major versions, run `npm rebuild`.

```bash
npm install
npm run build      # tsc → dist/
npm test           # vitest run
npm run typecheck  # tsc --noEmit
npm run dev        # tsx src/cli.ts  (re-runs the CLI without a build step)
```

There is currently no `lint` script; we rely on `tsc --strict` to catch the common stuff. A `npm run lint` step will land alongside the structured-comment-tag rule in spec 41 Wave B.

To run the CLI without installing it globally: `npx . start --stdio-only` or `node dist/cli.js start --stdio-only`.

## Testing philosophy

**Vitest, no live network in unit tests.** Every adapter takes a test-mode seam so the external command can be stubbed. For the GitHub adapter, that seam is `registerGithubTools(server, { exec })` — pass your own runner and assert on its inputs. See `tests/github-adapter.test.ts` for the pattern.

**Integration tests use the daemon.** Tests that exercise SSE fan-out or multi-client behavior start a real `mountTransports` against an in-memory SQLite store (`new EventStore().init(':memory:')`) and connect real `EventSource` clients to it. See `tests/manual/` for working examples.

**Test data is fixtures, not fakes.** When a test needs a stored event, it goes through `EventStore.appendEvent` rather than constructing a `StoredEvent` directly. This keeps the test surface close to production behavior.

**A failing test is a bug report.** Don't `.skip` a test to land a PR. If the test is wrong, fix the test in the same commit as the code change and say so in the message.

## Writing a new adapter

See [`docs/writing-an-adapter.md`](./docs/writing-an-adapter.md) for the full walkthrough, including a runnable weather example in `examples/weather/`. The short version:

1. Create `src/adapters/<name>.ts` that exports `registerXxxTools(server, opts?)`.
2. Inside, call `server.registerTool('<name>.<verb>', { description, inputSchema }, handler)` per tool. Use Zod for input validation.
3. Wrap the external call (`execFile` / `fetch`) so subprocess/HTTP errors are normalized into a `toolError(...)` (or richer typed-error response, like `ghErrorToTool`).
4. Take a test seam — accept an injectable runner in `opts` so tests can stub the external call.
5. Wire it into `createSwitchServer` in `src/server.ts`.
6. Add a test in `tests/<name>-adapter.test.ts` (mirroring `tests/github-adapter.test.ts`).
7. Update the README's adapter list.

## Writing a new event kind

1. Add the kind to the `EventKind` enum in `src/event-types.ts`.
2. Add a Zod schema for the payload (e.g. `WidgetCreatedPayload = z.object({...})`).
3. Wire it into `validatePayloadForKind` in the same file.
4. (Wave B, planned) re-run `npm run build` to regenerate `dist/schemas/events.json`.
5. Add a test that round-trips a representative event through `EventStore.appendEvent` + `getEvents`.

Payloads MUST be JSON-serializable — events go through SQLite as JSON text.

## Code style

- **ESM imports**, including the `.js` suffix on local imports (we compile to ESM and Node's NodeNext resolver wants it):

  ```ts
  import { Broker } from './broker.js';   // yes
  import { Broker } from './broker';      // no
  ```

- **Strict TypeScript**. No `any` in new code unless you can write a one-line comment explaining what the boundary is. Prefer `unknown` + narrowing for data crossing a trust boundary.

- **Don't add try/catch to make tests pass.** If a code path can fail, model the failure — return a structured error, throw a typed error class. The `gh` adapter's `GhExecError` is the pattern.

- **Prefer pure functions over classes**, unless the thing genuinely owns mutable state across method calls (Broker, EventStore). Adapters are stateless and don't need classes.

### Structured comment tags

We use five greppable prefixes for inline comments. They are optional but encouraged on exported functions and on any code where the behavior isn't obvious from the signature.

| Tag | Purpose |
|---|---|
| `// CONTRACT:` | An invariant the function must uphold for callers. |
| `// INVARIANT:` | An invariant the surrounding code maintains. |
| `// SIDE-EFFECT:` | A non-obvious external effect (DB write, kill a process, network call). |
| `// FAILURE-MODE:` | What happens on the unhappy path. |
| `// WHY:` | Rationale that isn't obvious from the code. |

Example:

```ts
// CONTRACT: caller must hold the broker's session lock before calling.
// SIDE-EFFECT: writes one row to the events table.
// FAILURE-MODE: throws if event.kind is not in EventKind.
async function publish(event: Event): Promise<StoredEvent> { ... }
```

Don't add tags where they would just restate the code. The bar is "would a reader who hasn't read this file before benefit?"

A future build-step (spec 41 Wave B) will surface these tags in the generated tool catalog and lint for missing top-level tags on public functions.

## Commit message conventions

Conventional commits, with a provenance tag for which agent wrote the commit:

```
<type>(<scope>): <subject> [cc-opus]

<optional body>
```

- Types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `ci`.
- Scope is optional but encouraged: `feat(transports): ...`, `fix(broker): ...`.
- Provenance tag at the end of the subject:
  - `[cc-opus]` — Claude Code (Opus model)
  - `[cc-sonnet]` — Claude Code (Sonnet model)
  - `[cc-opus-via-co]` — CC Opus driven by Co
  - `[co]` — Co directly
  - Omit for human-authored commits.

Keep the subject under ~72 chars. The body is for *why*, not *what* — diff already shows what changed.

For multi-commit PRs, commit per logical artifact (ARCHITECTURE.md is one commit, CONTRIBUTING.md is another) so review can advance one piece at a time.

## Pull request hygiene

- One spec wave per PR. If a spec splits into Wave A / Wave B, those are separate PRs.
- PR title mirrors the spec: `Spec NN Wave X — <short description>`.
- Body: a one-paragraph summary, then a `Test plan` checklist, then a `DO NOT MERGE` if the author wants review before merge.
- Don't merge your own PR unless the task explicitly authorized it. The default is "Kenneth reviews, Kenneth merges."

## Adding a new gated tool — don't bypass the scope check

If you're adding a CONFIRM-tier tool (gated by `await_decision`), call
`gatedAction` from `src/tools/gated-action.ts` and pass `scopeCheck: { tool,
args, trustStore }`. That's the only thing you need to wire up — the trust
scope short-circuit runs automatically, and the existing `await_decision`
path runs when no scope covers the call. Do **not** roll your own decision
flow; the scope check is centralised on purpose so that future improvements
to scoping (matcher language, channels, multi-user) land in one place.

The same rule holds for the `WorkerOrchestrator.gate` helper — gating goes
through it, and the trust-scope path is already inside.

## When you're stuck

Three places to look:

1. **The relevant spec** in `../privacy tracker/specs/`. The spec is the canonical "what should happen and why."
2. **The ADRs** in [`adr/`](./adr/). The ADRs are the canonical "why is this code shaped like this?"
3. **Recent commit messages** (`git log --oneline -20`). The `[cc-*]` tag tells you which agent wrote a given commit; the body tells you what they were trying to accomplish.
